# 根据审查结果修复代码

## 目标

将 code review 结论转为可落地修复，优先降低线上风险，并提供可追踪的验证与剩余问题说明。

默认策略：在不臆断业务语义、不破坏架构分层的前提下，尽量修复所有可安全自动修复项。

## 触发场景

- 用户要求“根据 review 结果修复”。
- 用户粘贴了带严重级别、文件位置的 review 结论。
- 用户要求根据 PR comments 批量修复问题。

## 输入要求

至少应包含以下信息（缺失时先补齐）：

- 问题位置（`path:line` 或可定位到函数/代码段）
- 问题描述与影响
- 严重级别（Critical/High/Medium/Low）

若业务规则不明确，必须先向用户确认，不得臆断。

## 修复流程

1. 归并问题清单：去重、合并重复项，按严重级别排序。
2. 确认修复边界：标出可直接修复项与需业务确认项。
3. 逐项修复：优先 `Critical`、`High`，再处理 `Medium`、`Low`。
4. 保持约束一致：
   - 不改变既有分层（`cmd -> server -> service -> biz -> data/impl`）。
   - 优先复用 `magicbox` 与 app 内现有能力，不重复造轮子。
   - 错误处理遵循项目既有模式（如 `merr`）。
5. 逐步验证：
   - 先做最小范围编译/lint/test。
   - 再做受影响模块验证。
6. 输出结果：给出变更清单、验证结果、未修复项与后续建议。

## 修复优先级与执行策略

- `Critical`：立即修复；若有多种方案，选最小风险改法。
- `High`：在同一轮修复；避免引入新行为变化。
- `Medium`：可安全修复则纳入当前轮。
- `Low`：低成本且确定性高时修复，否则记录为待办。

遇到以下情况应暂停自动修复并提问确认：

- 需要改动业务语义或接口契约（proto/API）才能解决。
- 需要跨模块重构且风险不可控。
- review 结论与当前代码不一致（疑似已变更或上下文不足）。

## 输出格式

按以下结构输出：

```markdown
# 修复结果

## 变更清单（按严重级别）
- [Critical|High|Medium|Low] `path/to/file`：一句话说明修复内容与原因

## 验证结果
- lint: [通过/失败 + 关键信息]
- test: [通过/失败 + 关键信息]
- build: [通过/失败 + 关键信息]

## 未修复项
- [级别] `path/to/file`：未修复原因（需业务确认/风险过高/上下文不足）

## 后续手工修复建议
- 对未修复项给出可执行步骤（尽量最小改动）
```

## 执行规则

- 一条问题对应一条修复动作或一条阻塞说明，不要混写。
- 不猜接口、不猜业务；不确定就确认。
- 禁止为“修复 review”引入无关重构。
- 新增 API 或修改 API 语义时，提醒同步 README（API 概览、用法）。
- 完成修复后必须提供验证结果；无法验证时明确说明原因与建议验证命令。

## 快速示例

```markdown
## 变更清单（按严重级别）
- [High] `app/goddess/internal/service/user.go`：为查询结果增加 nil 判断，避免 panic，并返回 `merr` 业务错误。
- [Medium] `magicbox/server/cron/cron.go`：补充启动失败错误向上返回，避免静默失败。

## 验证结果
- lint: 通过
- test: 通过（`go test ./...`）
- build: 通过（`make goddess`）

## 未修复项
- [Low] `app/goddess/internal/data/repo.go`：涉及索引策略，需要 DBA/业务确认后再改。

## 后续手工修复建议
- 先确认索引命中目标查询，再按 `EXPLAIN` 结果调整 where 条件与索引组合。
```
